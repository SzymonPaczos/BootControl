# BootControl GUI v2 — Accessibility Red-Team Review

**Reviewer persona:** senior a11y engineer, daily Orca user (GNOME 46), four GTK + two Qt shipped a11y projects. Tested against AT-SPI2 expectations, WCAG 2.1 AA/AAA, and `prefers-reduced-motion`.
**Inputs read:** `docs/UX_BRIEF.md` (full), `docs/UX_MAPPING.md` (full), `docs/GUI_V2_SPEC.md` (full, all 895 lines), `ROADMAP.md` Backlog §"Gothic 2 mode".
**Scope contract:** this review attacks the **v2 a11y baseline** (UX_BRIEF §8 — "launch criterion, not v2"). The Gothic-2 power keymap is backlog and out of scope here, but it is referenced as a forcing function: v2 must not paint itself into a corner that makes Gothic-2 impossible.

## 1. Top 3 critical a11y failures

### 1.1 The Confirmation Sheet's `accessible-description` is a single static string — diff, preflight, and live-job-log updates will be invisible to Orca

**Where:** `UX_BRIEF.md:115` ("Confirmation Sheet sets `accessible-role: dialog` and `accessible-description` to the restated target string"); `GUI_V2_SPEC.md:567` ("On polkit success, the sheet's lower half collapses, replaced by a `LiveJobLog` streaming the four commands' output").

**The failure:**
The brief promises one `accessible-description` set once, on open, equal to the "restated target string". But the sheet's content is *dynamic*:

1. Preflight checks transition `pending → running → pass | fail` (per the state machine `GUI_V2_SPEC.md:606-609`).
2. After polkit, "the sheet's lower half collapses, replaced by a `LiveJobLog`" — a brand-new region of content the user never heard.
3. The destructive button's `enabled` flips when type-to-confirm matches and preflight passes (`GUI_V2_SPEC.md:565`).
4. On polkit denial, an inline `InfoBar --error` appears mid-sheet (`GUI_V2_SPEC.md:567`).

What Orca will say: only the original description ("Replace Secure Boot platform key. Device /sys/firmware/efi/efivars/. Current PK 5C:A0…"). Then **silence**. The user cannot tell whether preflight is still running, whether their typed `REPLACE-PK` was accepted, whether polkit succeeded, or whether commands are streaming. They will tab to the destructive button, hit Space, and either nothing happens (still disabled and they don't know why) or the daemon runs and they have no audible signal it has started.

What the user expects: a continuous narration. "Preflight, ESP mounted, pass. Free space, pass. Snapshot directory writable, pass. Type-to-confirm field empty. Replace PK button, dimmed, disabled."

**Fix (mandatory for v2):**
- Spec the sheet as an `accessible-role: dialog` **container** with `aria-live="polite"` semantics on three nested regions: preflight list, type-to-confirm status, job log. In Slint this maps to `accessible-role: list` for preflight and `accessible-role: log` (if Slint supports it; otherwise `accessible-role: text` with `accessible-description` re-emitted on change) for the job log.
- Each `PreflightCheck` row must have its own `accessible-label` that includes its `state` ("ESP mounted at /boot/efi, passed"), and the row must re-emit when `state` mutates.
- The destructive button's `accessible-description` must include *why* it's disabled: "Replace PK, button, disabled. Type REPLACE-PK and wait for preflight to pass."
- §6 component table must add `accessible-description` and `accessible-role` columns to **every** component.

### 1.2 Enter cancels — but Slint propagates Enter as default-button activation; this collides with assistive tech that emits Enter for "activate focused control"

**Where:** `UX_BRIEF.md:82` ("Hitting Enter cancels"); `GUI_V2_SPEC.md:567` ("Enter and Esc both activate Cancel"); keyboard map `GUI_V2_SPEC.md:790`.

**The failure:**
Orca, BRLTTY, and AT-SPI's keyboard-emulation paths all assume Enter activates the *focused* control. The brief says the Cancel button is the "bold default" — meaning *visually* default. If Slint binds Enter at the window level to fire `cancel()`, fine. But:

- A screen-reader user tabs to the **destructive** button. The button has focus. They press Space (per `UX_BRIEF.md:114` — "primary button activated by Space"). OK.
- Same user, different muscle memory: presses Enter on the focused destructive button. Orca emits the keystroke, AT-SPI dispatches it, the focused button receives it (because button widgets handle Enter natively in every toolkit including Slint) — the destructive op fires.
- *Or*, depending on Slint's event ordering, the window-level Enter handler fires first and cancels. Either way, the user cannot predict which.

There is no documented Slint API to "swallow Enter at the window level even when a focused widget would consume it" — and even if there were, it would surprise a user who tabs to the destructive button expecting button-style activation.

The brief's reasoning ("safety: Enter cancels so panicked users back out") works for mouse users muscle-memory-pressing Enter. It is **broken** for keyboard-only users, who reach the destructive button by intent — `Tab → Tab → Tab` — and must then press Space because Enter is sabotaged. That's a discoverability cliff: Space-not-Enter on buttons is non-standard everywhere except the brief.

**Fix (mandatory for v2):**
- Decouple "Enter at window level" from "Enter on focused button". Spec it explicitly: "Enter, when the *type-to-confirm input* is the focused widget, has no destructive effect — it is consumed but does not activate. Enter, when *any button* is focused, activates that button. Enter, when *no button is focused* (focus on body text or the diff), invokes Cancel."
- Add an `accessible-description` to the destructive button: "To activate, press Space. Pressing Enter does nothing." This is hostile copy — propose instead: drop the "Enter cancels" rule globally and rely on (a) destructive button being last in tab order, (b) Cancel being the visual default, (c) destructive being disabled until type-to-confirm.
- See §7.1 for the question-by-question response.

### 1.3 Live system state changes (daemon disconnects, snapshot completes, polkit prompt appears) have no spec'd live-region announcement

**Where:** `UX_BRIEF.md:122` ("control enters a `pending` state"); `GUI_V2_SPEC.md:108` (Sidebar shows `daemon: ●`); `GUI_V2_SPEC.md:428` ("Logs tab badge in the sidebar shows a `●` indicator"); state machine `GUI_V2_SPEC.md:626-635`.

**The failure:**
Five state changes that affect the entire app must be announced to a screen reader. None are spec'd as `aria-live` / live-region equivalents:

1. **Daemon disconnect.** The status dot in sidebar (`GUI_V2_SPEC.md:99`, "daemon: ●") flips to red. Orca says nothing. The user keeps clicking buttons that silently no-op.
2. **Polkit prompt appears.** Polkit-gnome opens its own window — this *should* be announced by the desktop, but if the polkit agent is running over Wayland with broken AT-SPI bridges, the prompt appears and BootControl's window loses focus without telling the user. UX_BRIEF §7 says "GUI shows a Pre-flight Card running checks" then "the daemon's privileged method" — between those two, focus moves to a different process. The sheet must announce: "Authorization requested. Polkit prompt opening."
3. **Streaming job badge in sidebar (`GUI_V2_SPEC.md:428`).** A blinking dot appears on Logs while a job streams. Visual-only. Orca will not announce until the user navigates back to the sidebar.
4. **InfoBar appears mid-flow.** A page in `staged` state suddenly gets a `--warning` InfoBar at the top because a precondition broke (e.g. `efivarfs` re-mounted read-only). The state machine doesn't model when InfoBars enter — and if the InfoBar `accessible-role: status` fires AT-SPI's `Object:StateChanged:showing`, Orca will announce only if the InfoBar is *focused* or has `aria-live`. Slint's support for `aria-live` polling is unclear (see §4 unknowns).
5. **`pending` opacity-50 spinner toggle (`UX_BRIEF.md:122`).** Visual-only. A screen-reader user toggles a checkbox, hears nothing, and doesn't know whether the daemon accepted it.

**Fix (mandatory for v2):**
- Spec a global `accessible-role: status` region in `appwindow.slint` whose `accessible-description` is updated on every state-machine transition. This is the *single source of truth* for screen-reader announcements: daemon status, page state ("editing", "staged 3 changes", "applying"), job-stream activity.
- Sidebar dot: must have `accessible-label: "Daemon connected"` / `"Daemon disconnected"` and re-emit on transition; same for the streaming-job badge.
- Confirmation Sheet: announce "Polkit prompt requested. Focus may move to authorization dialog." just before invoking the daemon's privileged method.
- Pending toggles: the staging transition must update an `accessible-description` on the changed control: "Default entry, Linux Mint, change staged. Apply or Discard from action footer."

## 2. Focus management audit

The brief is silent on focus management almost everywhere. This is the #1 a11y bug-source in modal apps and it has not been designed.

### 2.1 Confirmation Sheet (the critical one — `GUI_V2_SPEC.md` §4, lines 482-567)

Walking through every focus-relevant moment:

- **On open:** spec doesn't say. Default Slint `PopupWindow` behaviour is to leave focus on the click target, which for keyboard users means **focus stays on the page button that opened the sheet, behind the sheet**. Orca will read the sheet's accessible-description on open (per spec) but Tab will move through the *page behind*, not the sheet — until the user finds out by trial. **CRITICAL fix needed:** spec must mandate `forward-focus` (Slint property) onto the sheet container or the type-to-confirm input on open.

- **Initial focus target:** The type-to-confirm input is a logical first stop *only when type-to-confirm is required* (PK replace, MOK enroll, key erase). For sheets without type-to-confirm (NVRAM backup, snapshot restore, GRUB rewrite without typed-confirm), where does focus land? Spec says nothing. Recommend: Cancel button (matches "Cancel is the bold default" — Cancel keyboard-default = focus-default on entry).

- **Tab order inside the sheet:** spec'd at `GUI_V2_SPEC.md:792`: "type-to-confirm → Cancel → Destructive". Two problems:
  1. The diff preview, command list, preflight checks, snapshot statement, and recovery line are all skipped. A screen-reader user who tabs from type-to-confirm straight to Cancel never reaches the diff. They cannot review what they're about to confirm. Tab order must include them as `accessible-role: text` stops, OR the sheet must set `accessible-description` to a fully-narrated body that includes the diff content as text (impractical for long diffs).
  2. "Cancel before Destructive" is correct for proximity. But it makes the destructive button the *last* tab stop. If focus then wraps (Tab again), it lands... where? Back on the diff preview? Back on type-to-confirm? Spec doesn't say. **CRITICAL:** spec must mandate `accessible-role: dialog` with **focus trap** — Tab from the destructive button cycles back to the first focusable element *inside the sheet*, never escaping.

- **Backdrop click:** `GUI_V2_SPEC.md:567` ("backdrop captures clicks"). Good for mouse. But what does keyboard equivalent look like? If a user accidentally Tabs into the dimmed page behind, can they Tab back? With a focus trap (above), no escape — good. Without one, they're lost.

- **On close (Cancel pressed):** spec doesn't say. Focus should return to the element that opened the sheet (the `[Apply…]` button on the action footer or the `[Delete]` button on the row). Slint's default behaviour after a `PopupWindow` closes is undocumented in this respect. **Fix:** spec must mandate "on close, focus returns to the element that opened the sheet."

- **State-machine transition `confirming → preflighting`:** preflight runs. Where is focus? Still on destructive button (which is now disabled because preflight is running). Orca says nothing about preflight progress. Spec must require focus to move to the preflight region OR fire a live-region announcement per check completion.

- **State-machine transition `authorizing` (polkit prompt):** focus leaves the BootControl window. When the polkit dialog closes, focus may return to BootControl or may go elsewhere depending on the compositor. Spec must require BootControl to grab focus back to the sheet on the polkit-success / polkit-fail signal.

- **State-machine transition `applying`:** sheet's lower half is replaced with `LiveJobLog`. Where is focus? If it was on the destructive button (now hidden), Slint may park it on the window root — meaning Tab is lost. Focus must move to the LiveJobLog region (which should be `accessible-role: log`).

- **State-machine transition `applied`:** sheet closes. Focus must return *not* to the original opener (which may no longer make sense — `[Apply…]` is gone because there are no pending changes) but to a sane next stop. Recommend: the page header h1 (so the next thing the user hears is the page title, confirming the world.).

- **State-machine transition `failed`:** sheet stays open; an `InfoBar --error` appears inside it (`GUI_V2_SPEC.md:629`). Where does focus go? Should go to the InfoBar's `[Retry]` button, to give the user the next-action affordance.

- **InfoBar mid-sheet:** spec doesn't model whether the InfoBar is *inside* the sheet's tab cycle or above it. If above, sheet's focus trap excludes it — broken. Must be inside.

**Severity:** CRITICAL. The Confirmation Sheet is the highest-stakes UI in the app. Lacking explicit focus-management spec means each of PR4 and PR5 will invent its own, inconsistently.

### 2.2 Other modals / sheets / popups

- **`?` shortcut sheet (`GUI_V2_SPEC.md:797`).** "Non-modal `PopupWindow` (`?` to open, `Esc` to close) so it can coexist with a Confirmation Sheet without nesting modals." If it's non-modal *and* a Confirmation Sheet is open, where does focus go when `?` is pressed? Stack management for screen-reader users is hostile here — Orca will announce both regions and the user cannot distinguish. Recommend: while a Confirmation Sheet is open, `?` is bound to "show shortcuts in the sheet's diff area" — not a separate popup.

- **Toast (`GUI_V2_SPEC.md` toast, §5 UX_BRIEF).** 4-second auto-dismiss bottom-right. For Orca users, 4 seconds may not be enough — Orca's verbose verbosity reads "Default entry changed to Linux Mint twenty one point three. Toast. Dismissible. Undo." in roughly 5–7 seconds. The user cannot get to the [Undo] button before it disappears. **Fix:** spec must require the toast to use the user's screen-reader timing (`prefers-reduced-motion` is the closest signal — when set, dismiss is disabled; user must dismiss manually). Also, toast must accept Tab focus (`accessible-role: alert` plus `accessible-description` on action button).

- **Polkit prompt.** Out-of-process. Not BootControl's responsibility — but BootControl must announce that it is requesting authorization (see §1.3 above) and must request focus back when the prompt resolves.

- **`[+ New entry]` inline expansion (`GUI_V2_SPEC.md:186`).** Says "NOT a modal" but "in-line panel". This is good for avoiding modal-on-modal. But it's a brand-new region of focusable controls. Spec must:
  - Move focus into the panel on expand.
  - Place an `accessible-role: region` with `accessible-label: "New entry form, expanded"`.
  - On collapse (Cancel inside), return focus to `[+ New entry]` button.

- **Inline rename on Boot Entries row (`GUI_V2_SPEC.md:182`).** F2 opens an inline `StyledInput`. On Esc/Enter, focus returns where? Must be the renamed row's title cell, not the toolbar.

- **Snapshot manifest expansion (`GUI_V2_SPEC.md:372`).** Click `▸ View manifest` expands inline. Same set of questions; spec is silent.

- **Advanced disclosures (`GUI_V2_SPEC.md` Bootloader page, §3.3, lines 237, 239, 251).** Multiple per page. Each is a focus-management hazard. See §4 below.

### 2.3 InfoBar mid-flow

UX_BRIEF §5 ("Persistent InfoBar — non-dismissible while the condition holds"). What happens when an InfoBar appears while the user has focus inside a card on the same page?

- The InfoBar appears above the cards. Focus does *not* move — correct, that would be hostile.
- Orca must announce. Slint's `aria-live` equivalent for newly-appearing widgets is **unknown** (see §4). If the InfoBar lacks `aria-live: assertive`, the user finishes typing into a chip and only finds out about the warning when they Tab up to it.

**Fix:** spec must require the InfoBar to fire an `accessible-description` change at the window-level live-region the moment it materialises.

## 3. Keyboard navigation audit

### 3.1 The keymap (spec §8) — chord-by-chord conflict review

| Chord | BootControl action | Conflicts with |
|---|---|---|
| `Tab` / `Shift+Tab` | Focus next/prev | None inherently. But Slint's tab-order computation when controls appear/disappear (e.g. action footer materialising) is not specified — see §3.3. |
| `Ctrl+1..6` | Sidebar jump | **Conflicts** with GNOME Terminal's tab-switching (Ctrl+1..n switches terminal tabs). Power-user sysadmins running BootControl from a terminal-spawned context lose tab-switch when BootControl has focus. Acceptable if they tab away first. **Conflicts** with Orca? No — Orca uses Insert+keys on default keymap, so Ctrl+digit is clear. **Conflicts** with GNOME Activities? No (Super-based). **Verdict:** acceptable but document the Terminal-switch overlap. |
| `Ctrl+,` | Settings | **Conflicts** with most GNOME apps' standard Preferences chord — *agrees* with them in fact, this is the right binding. |
| `↑/↓` on sidebar | Move selection | None. |
| `Enter`/`Space` on sidebar | Activate | None. |
| `Ctrl+S` on page with footer | Apply | **MAJOR CONFLICT.** See open question #7 below. (a) Mac muscle memory ("Cmd+S") — irrelevant on Linux. (b) Text-editor mental model: "Save" is reversible; this Apply triggers polkit, snapshots, and a daemon write. Mismatch. (c) **Conflicts** with the in-app text editor (`GUI_V2_SPEC.md:212` "[Open as raw text]" — when raw editor is focused, Ctrl+S should save the textarea contents to staging, not trigger Apply). The spec doesn't disambiguate. (d) **Conflicts** with Orca's "say all from here" flat-review chord on some Orca configs (Orca-default uses Insert+chords, but power users rebind). **Verdict:** rebind to `Ctrl+Return` or require an action-footer focus before Ctrl+S works. |
| `Esc` on page with footer | Discard | **CONFLICT** with Esc inside an open chip-add input — the user is typing a parameter, presses Esc to cancel the *chip add*, and instead **discards all staged changes**. Catastrophic for a sysadmin who has carefully staged 5 cmdline parameters. Spec must scope: Esc on the *page background* discards; Esc inside an input cancels the input and stops there. |
| `Ctrl+↑/Ctrl+↓` on row | Reorder | **CONFLICT** with GNOME Mutter's workspace-switch on some configs (Ctrl+Alt+Up/Down is workspace-switch by default; Ctrl+Up/Down without Alt is not bound by Mutter — checked GNOME 46 defaults — so this is **safe**, but document it). **CONFLICT** with text-editor "extend selection by line" semantics — but boot-entries rows are not text editors, so contextually fine. |
| `F2` | Inline rename | None — universal. |
| `Delete` | Delete entry | **CONFLICT** with text-input forward-delete when an inline-rename input has focus. Spec must scope: Delete on a focused row triggers Delete sheet; Delete inside an input is forward-delete. |
| `Enter` in Confirmation Sheet | Cancel | See §1.2 — broken contract. |
| `?` | Shortcut sheet | **CONFLICT** with type-to-confirm input — if the user is typing `REPLACE-PK?` into a future hypothetical typed-confirm token, `?` triggers the popup instead. Spec scope: `?` only triggers when no input has focus. |
| `Ctrl+C` on LiveJobLog | Copy log | **CONFLICT** with terminal-style "interrupt" mental model. Power users may expect Ctrl+C to *cancel the running job*. Spec doesn't define cancel-job behaviour at all. Either rebind to Ctrl+Shift+C, or implement Ctrl+. (period) as cancel and Ctrl+C as copy. |
| `Ctrl+F` | Logs search | None. Universal. |

### 3.2 Missing keyboard contracts

The spec does not specify keyboard equivalents for several mouse-only interactions:

- **Backend chip click (`GUI_V2_SPEC.md:111-114`).** "Click backend chip — navigates to Bootloader page." How is a chip focused via keyboard? It's not in the §8 keymap. Must be Tab-reachable; Enter activates; spec must say so.
- **Inspector field click-to-copy (`GUI_V2_SPEC.md:188`).** "Click an Inspector field … copies to clipboard." Keyboard equivalent? Spec is silent. Must be Ctrl+C when an Inspector field is focused, with `accessible-description: "Press Ctrl+C to copy."`.
- **`▸ View manifest` (`GUI_V2_SPEC.md:372`).** Disclosure expand/collapse on snapshots. Keyboard? Universal pattern: Space when focused. Spec must require it.
- **Right-click `[Copy timestamp]` (`GUI_V2_SPEC.md:375`).** Right-click is mouse-only. Must have a keyboard equivalent — context-menu key (Shift+F10 or the dedicated `Menu` key).
- **`[Show hidden ☐]` toggle (`GUI_V2_SPEC.md:152`).** Checkbox; Space activates when focused. Standard. But spec doesn't say it's in tab order. Implicit yes; make it explicit.
- **Action footer `[Apply…]` and `[Discard]` order (`GUI_V2_SPEC.md:169`).** "Discard left, Apply right." Tab order should match visual order. UX_BRIEF §6 step 3 says Cancel left, primary right — same shape, fine. But the spec must require Apply *not* be reachable by Enter when the footer just materialised — newly-appeared default buttons trap users mid-edit.

### 3.3 Action footer materialisation = focus surprise

When `staged-count` flips from 0 to 1, the `ActionFooter` materialises (`GUI_V2_SPEC.md:169`, §3.2). Two failure modes:

1. **Tab order shifts mid-flow.** A user has just Tab'd through Boot Entries and is at "Show hidden" toggle. They hit Tab expecting to land on the next row — but now ActionFooter has appeared between row N and the next list region, so they land on `[Discard]` instead. Disorienting.
2. **Focus might jump to the Apply button.** If Slint considers a newly-appeared "default" button focusable and the framework's default is "focus the new default", a screen-reader user is yanked from their typing context to "Apply 1 change, button". Spec must explicitly **forbid** auto-focus on ActionFooter materialisation.

## 4. Screen-reader semantics — `accessible-role` / `accessible-label` audit

Slint's accessibility model (as of Slint 1.5+ — confirm against runtime version) supports:
- `accessible-role` enum: `none`, `button`, `text`, `text-input`, `check-box`, `combo-box`, `slider`, `tab`, `tab-list`, `tab-panel`, `progress-indicator`, `list`, `list-item`, `groupbox`. **Notable absences:** `dialog`, `log`, `status`, `region`, `alert`, `tooltip`, `menu`, `menubar`, `tree`, `treeitem`, `toolbar`. (Cite: Slint docs `Accessibility` page; verify in source.)
- `accessible-label`, `accessible-description`, `accessible-value`, `accessible-checked`, `accessible-positional` properties — varying support.
- `accessible-action-default` — programmatic activation.

**Per-component audit:**

| Component (§6) | Spec'd role | Spec'd label | A11y verdict |
|---|---|---|---|
| `Sidebar` | none in spec | none | **Fail.** Must be `accessible-role: list` (Slint supports). Each item: `list-item` with `accessible-label` = page name + ", " + count of pending changes if > 0. The "daemon: ●" dot must be a focusable status with `accessible-label: "Daemon status: connected"`. |
| `ActionFooter` | none in spec | none | **Fail.** No matching Slint role for "footer" or "toolbar". Best fit: `accessible-role: groupbox` with `accessible-label: "Action footer, 1 change pending."`. Buttons inside use `button` role natively. |
| `ConfirmationSheet` | spec'd as `dialog` (UX_BRIEF §8) | `accessible-description` = restated string | **Slint cannot express `dialog` role.** This is a hard limitation. Workarounds: (a) approximate with `accessible-role: groupbox` + `accessible-label: "Dialog: Replace Secure Boot platform key"`; (b) file an upstream Slint issue and gate v2 GA on resolution; (c) wrap the sheet in a native windowing-system dialog if we can't fix Slint. Engineering must verify Slint's current support before PR4 merges. **Block PR4 close on Slint dialog support.** |
| `InfoBar` | none in spec | none | **Fail.** Best fit: `accessible-role: groupbox` + `accessible-label: "<severity>: <message>"`. The severity must be in the label, not just the colour, per WCAG 1.4.1 (use of colour). Confirm Slint can re-emit `accessible-description` to act as a live region; if not, this is an unknown — see §8 below. |
| `Toast` | none | none | **Fail.** Slint has no `alert` role. Use `groupbox` + label "Notification: <message>". Auto-dismiss at 4 s is incompatible with screen reader users — see §2.2. |
| `DiffPreview` | none | none | **Fail.** A diff is a structured artefact; flat text is hostile. Best Slint mapping: `accessible-role: list` of lines, each line a `list-item` with `accessible-label: "Removed line: <text>"` or `"Added line: <text>"` (severity in the label, not colour). The `+` and `-` glyphs must NOT be the only differentiator — WCAG 1.3.3 (sensory characteristics). |
| `Inspector` | none | none | **Fail.** Each field row should be `accessible-role: text` with `accessible-label: "<key>: <value>"` and `accessible-description: "Press Ctrl+C to copy <value> to clipboard."`. |
| `PreflightCard` | none | none | **Fail.** Must be `accessible-role: list`. Each `PreflightCheck` row: `list-item` with `accessible-label: "<check label>, <state>"` and re-emit on `state` change. The `◯ ● ✕` icons are colour+glyph; pass WCAG 1.4.1 because of the glyph, but only if `accessible-label` carries the state in words. |
| `LiveJobLog` | none | none | **Fail.** No Slint `log` role. Best fit: `accessible-role: text` with `accessible-multi-line: true` (if available). The autoscroll behaviour is hostile to screen readers — when autoscroll fires and content shifts, Orca's review cursor loses position. Spec must add an "Autoscroll" toggle, default OFF for screen-reader users (detect via `getenv("ORCA_RUNNING")` is unreliable; best is to expose the toggle and announce it). **Critical:** when the job emits a fail line (exit ≠ 0), the failure must be announced — not just the exit-code field — via a live-region update. |
| `PrimaryButton` / `DangerButton` / `GhostButton` | implicit `button` | `text` from `in text` | OK if `accessible-label = text` is verified. DangerButton must additionally carry `accessible-description: "Destructive action."` so Orca says "Replace PK, button, destructive action" — pre-warning the user before they hit Space. |
| `StyledInput` | implicit `text-input` | none | **Fail on label.** Every input must have an `accessible-label` from a parent form-row label. Spec must require label-input pairing. |
| `SpinBox` | `slider`? `text-input`? | none | **Unknown.** Slint has no native `spin-button` role. Without it, Orca will say "5, slider, 0 to 60" — not "5 seconds, GRUB timeout". Need `accessible-role: text-input` (so the user can type the value) plus `accessible-label: "GRUB timeout in seconds"` and `accessible-value: "5"` re-emitted on change. |
| `ParamChip` | none | none | **Fail.** Each chip is a button-like artefact with text + a remove control. Best mapping: a `groupbox` containing `list-item` ("kernel parameter `quiet`") + `button` ("Remove parameter `quiet`"). Slint may struggle with this nesting; verify. |
| `Card` | none | none | **Fail.** Use `accessible-role: groupbox` with `accessible-label = title` and `accessible-description = description`. |

**Slint-specific unknowns to verify before PR4 merges:**

1. Does Slint emit AT-SPI `Object:StateChanged:showing` events when widgets become visible (e.g. ActionFooter materialising)? If not, screen readers won't know the footer appeared.
2. Does `accessible-description` re-emission fire AT-SPI `Object:PropertyChange:accessible-description` events? If not, our "live region" approach is broken — see §8.
3. Does Slint propagate `prefers-reduced-motion` from GNOME's `org.gnome.desktop.interface`? If not, animations (motion-fast/medium/slow tokens) violate WCAG 2.3.3.
4. Does Slint's `PopupWindow` set its AT-SPI parent relation correctly so Orca knows it's modal? If not, Orca treats it as a peer of the main window — focus reads incorrectly.
5. Does Slint support keyboard focus trapping inside a `PopupWindow`? If not, we need to manually intercept Tab — and we can't, because Slint's focus-handling is not exposed at the application level.

**Verdict:** §6 of the spec must add a row "accessible-role" and a row "accessible-label source" to the component-API table, and PR4 must be gated on verifying the five unknowns above. Without those answers, the v2 a11y baseline has no foundation.

### 5.1 Color contrast — Catppuccin Mocha against UX_BRIEF §4 token table

1. **`--on-surface-disabled` (`#6c7086`) fails AA for normal text on `--surface`.** UX_BRIEF doesn't specify where this token is used. If used for "disabled" labels or the destructive button when preflight hasn't passed, a low-vision user cannot read what the button is for. Either bump to a lighter mocha overlay (`overlay1` `#7f849c` ratio ~4.6:1 borderline) or accept that disabled controls only need 3:1 (WCAG SC 1.4.11 inactive-component exemption). Document the choice.

2. **Button label colour is undefined.** Spec must add: `PrimaryButton` label = `--surface`; `DangerButton` label = `--surface`; `GhostButton` label = `--on-surface`. Current default is implied as `--on-surface`, which gives a 1.4:1 catastrophe.

3. **Diff preview `+ green` / `- red` lines on `--surface-container-high`.** `--success` `#a6e3a1` on `#11111b`: ~12.0:1 PASS. `--error` `#f38ba8` on `#11111b`: ~7.5:1 PASS. Good.

4. **Focus ring `2 px --accent at 60% alpha` on `--surface-container-high`.** Effective colour: roughly `#cba6f7` × 0.6 + `#11111b` × 0.4 = approx `#7c659a`. Against `--surface-container-high` (`#11111b`): ~2.7:1. **FAIL WCAG 2.1 SC 1.4.11 (Non-text contrast 3:1 minimum)**. The focus ring is invisible-ish to low-vision users on the darkest card. **Mandatory fix:** either (a) raise alpha to 100% (`--accent` directly), or (b) double the ring thickness to 3 px and add an inner 1 px `--surface-container` separator for definition. UX_BRIEF §8 must be amended.

5. **High-contrast mode is mentioned (`UX_BRIEF.md:116`) but not specified.** No tokens for HC dark or HC light. PR1's `tokens.slint` ships only Mocha. The brief promises "token table is the single source of truth; high-contrast mode swaps the Mocha map for a higher-contrast variant". This is a future-tense promise. Spec must add a stub `tokens-high-contrast.slint` to PR1 even if empty, otherwise PR5 closes "v2" without HC mode and we ship a broken accessibility commitment.

### 5.2 Focus rings — visibility audit

Beyond contrast, the focus ring is spec'd at "2 px outline" (`UX_BRIEF.md:114`). Concerns:

- **Outside-the-element vs inside.** Spec doesn't say. If inside, controls with the focus ring shrink their content area. If outside, neighbouring widgets must allow 2 px of headroom — Slint's layout doesn't auto-pad. Each component needs an explicit `padding-when-focused` or the focus ring will clip neighbouring controls. Document.
- **Cards have rounded corners (`radius-large: 12px`).** Focus ring must follow the radius. Verify Slint's outline behaviour respects border-radius — if not, focus rings clip on rounded controls.
- **Focus ring vs `--accent` button fill.** When the `PrimaryButton` (which is `--accent` filled) has focus, its focus ring is also `--accent` — invisible. Must use a different focus colour on accent-filled buttons (`--on-surface` would work if the inside is `--surface` — see §5.1). Spec must add: "for accent-filled controls, focus ring is `--on-surface` instead."

### 5.3 Motion — `prefers-reduced-motion`

UX_BRIEF mentions `motion-fast: 120ms`, `motion-medium: 180ms`, `motion-slow: 300ms` (`tokens.slint`, `GUI_V2_SPEC.md:753-755`). No mention of `prefers-reduced-motion`.

**Failure:** WCAG 2.3.3 requires honouring this setting. Sheet open/close animations, ActionFooter materialise, Inspector slide-in, toast slide-up — all violate if not gated.

**Fix:** `tokens.slint` must include a `motion-multiplier` (1.0 default, 0.0 when `prefers-reduced-motion`). Slint's support for reading the system pref is **unknown** — verify upstream. If unsupported, expose a Settings toggle "Reduce motion" and document in known-limitations.

### 5.4 Font-size scaling

UX_BRIEF §4 uses fixed `px` units. At 200% scale (`gsettings set org.gnome.desktop.interface text-scaling-factor 2`), Slint applications **may or may not** respect the system scaling factor — Slint has its own scaling. Verify; document; if Slint doesn't honour text-scaling-factor, the spec's "Font size" Settings dropdown (`GUI_V2_SPEC.md:457`) is the only workaround, and it must offer at least 100/125/150/200% steps.

## 6. State machine a11y (`GUI_V2_SPEC.md` §5)

The pending-changes machine has 9 states. None has an `accessible-description` mapping. Walking each transition:

| Transition | Visual signal | Required announcement | Live region |
|---|---|---|---|
| `clean → editing` | input gains focus | Implicit via Orca "<label>, edit text". OK if `accessible-label` is set. | none |
| `editing → staged` | ActionFooter materialises | "Footer appeared. 1 change pending." — **not currently spec'd**. | window-level live region |
| `staged → staged` (more edits) | counter increments | "Footer updated. 3 changes pending." — must be debounced (300ms) so each keystroke doesn't re-announce. | yes |
| `staged → clean` (Discard) | Footer collapses | "Changes discarded. Page is clean." | yes |
| `staged → confirming` (Apply pressed) | sheet opens | Sheet's `accessible-description` reads. Focus moves into sheet. | dialog open event |
| `confirming → preflighting` | preflight runs | Per-check announcements. See §1.1. | yes — preflight list region |
| `preflighting → blocking-error` | InfoBar inside sheet | "Preflight failed. ESP not mounted. Mount the ESP and retry." Focus to InfoBar's [Retry]. | yes |
| `preflighting → authorizing` | polkit prompt fires | "Authorization requested. Polkit prompt opening." | yes |
| `authorizing → applying` | sheet lower half collapses, LiveJobLog appears | "Applying. Streaming command output." Focus to LiveJobLog. | yes |
| `applying → applied` | sheet closes, InfoBar success | "GRUB rewritten. 4 keys changed. Snapshot saved. Page is clean." Focus to page header. | yes — page-level live region |
| `applying → failed` | sheet stays, InfoBar error | "Apply failed. Exit code 1. Snapshot kept. Press R to retry, C to cancel." Focus to InfoBar. | yes |

**Observations:**

1. The state machine spec has zero a11y annotations. PR4 must add a column "Announcement" or "Accessible state description" to the state-machine documentation.
2. The `applied → clean` transition is the most subtle: the sheet closes, the page beneath comes back. If the sheet's success InfoBar replaces the sheet's content for 1 s before close (Cockpit pattern), screen-reader users have time to hear it. If it closes immediately and an InfoBar appears on the page, the user's focus may already have moved — they'll miss the announcement. Recommend: keep the sheet open for 2 s post-success showing the InfoBar inside it, then close.
3. **The `failed` state is the most a11y-critical** because the user is in a bad place and needs orientation. Spec must require: focus to [Retry] (named action), `accessible-description` includes the exit code AND the snapshot timestamp ("Snapshot 2026-04-30T13:14:22-grub-rewrite saved; you can restore from Snapshots page if needed").

## 7. Per-flagged-question attack

### 7.1 Q1 (`GUI_V2_SPEC.md:881`): "Enter cancels — too hostile to keyboard-only?"

**Yes, it's broken — see §1.2.** The brief's Enter-cancels rule is a single-handed solution to a two-handed problem. It optimises for "panicked mouse user pounding Enter" and pessimises "intentional keyboard user planning a 50-machine key rotation".

**Concrete attack:** even with mouse users, Enter-cancels fails when the user has Tab'd into the type-to-confirm field, typed `REPLACE-PK`, and presses Enter expecting to commit (universal text-input mental model). The field swallows Enter, no destructive action, the user types Enter twice, third time they Tab to the destructive button and press Space — friction, but no harm. So the rule mostly inconveniences keyboard users and never actually saves a panicked mouse user (they'd hit Esc, not Enter).

**Recommendation:**
- Drop the Enter-cancels rule.
- Rely on (a) Cancel being the visual default (bold, left), (b) destructive button being last in tab order, (c) destructive button being disabled until type-to-confirm + preflight, (d) all destructive buttons explicitly labelled as such in their `accessible-description`.
- Treat Enter as standard: activate the focused control. Hold the line on Esc-cancels everywhere.

### 7.2 Q4 (`GUI_V2_SPEC.md:887`): "One disclosure or two for [Erase enrolled keys]?"

A disclosure is a focus-management hazard regardless of depth. From an a11y standpoint:

- **One disclosure + type-to-confirm** is fine *if* the disclosure's expanded state is announced ("Strict mode disclosure, expanded") and the destructive button inside is marked clearly ("Erase enrolled keys, button, destructive action — irreversible").
- **Two disclosures** doubles the focus-management burden without adding meaningful safety for screen-reader users. They tab through both disclosures regardless; the second disclosure adds 2 Tab stops with no value to a sighted novice either (they'll click through both).
- A *better* gate is **per-action authentication ceremony** — requiring the user to type a longer phrase ("ERASE-ALL-KEYS-I-UNDERSTAND") rather than adding another Tab stop. Type-to-confirm scales naturally to severity.

**Recommendation:** keep one disclosure; raise the type-to-confirm string for `[Erase enrolled keys]` to a longer phrase. A11y-wise, that's lower-friction than nested disclosures.

### 7.3 Q7 (`GUI_V2_SPEC.md:893`): "`Ctrl+S` collision"

Already attacked in §3.1. Two specific a11y concerns beyond the open question's framing:

1. **Orca's "say all from here" on some configs uses Ctrl+;**, but power users rebind to Ctrl+S in vim-like setups. Linux a11y users with non-default Orca keymaps can collide. Not common but real.
2. **Slint's keyboard accelerator dispatch may not respect AT-SPI's keystroke listeners.** If Orca consumes Ctrl+S for "say all" and Slint also fires the binding, the user experiences both — log applied AND screen reader narrating. Confusing.

**Recommendation:** rebind to `Ctrl+Return` (the GNOME pattern for "commit message" in many apps, e.g. GNOME Builder). It's intentional, two-key, and unambiguous. Keep `Ctrl+S` as an alias only if extensive sysadmin user-testing shows it's expected.

### 7.4 Q8 (`GUI_V2_SPEC.md:895`): "InfoBar 8 s vs persistent"

For screen-reader users, **8 s is sometimes enough, sometimes not** — depends on Orca verbosity, message length, and whether the user is mid-sentence on another keyboard action. Auto-dismiss based on a fixed timer is hostile to a11y.

**Concrete failure:** "GRUB rewritten · 4 keys changed · snapshot 2026-04-30T13:14:22-grub-rewrite saved" is approximately 11–13 seconds for Orca at default verbosity (44 syllables). The InfoBar disappears mid-narration. The user hears "GRUB rewritten · 4 keys changed · snapshot 2026-04—" then the live region clears. They never hear the timestamp.

**Recommendation:**
- Change from "8 s auto-dismiss" to "auto-dismiss when the user takes ANY interactive action OR when 30 s pass (whichever first)". This is the GNOME `AdwToastOverlay` pattern.
- Add a Settings option "Notification persistence" with values [Auto / Always wait for dismiss] — default Auto, a11y users flip to "Always".
- The InfoBar must be dismissible by keyboard (Esc when focused), and Tab must reach it.

## 8. New a11y questions the designer didn't think to ask

1. **Does Slint emit AT-SPI announcements when `accessible-description` changes on an existing widget?** If not, our entire live-region strategy collapses — preflight progress, daemon-status changes, staged-count updates all become silent. **Block PR4 close on a verified test.** If unsupported, file an upstream Slint issue and add a workaround (focus-cycle the live region to force re-announce; ugly, but works on most ATs).

2. **Does Slint's `PopupWindow` correctly mark itself as modal in AT-SPI?** Modal dialogs require `Object:state:modal=true` and the parent window's `Object:state:active=false`. Without this, Orca lets users navigate to the dimmed parent — and reads it. Verify.

3. **How does Slint translate keyboard focus across Wayland compositor transitions?** When polkit-gnome opens its prompt, Slint's window loses focus. When polkit closes, where does focus return? Spec must require BootControl's daemon-success signal handler to call `set-focus(<sheet's primary action>)` explicitly.

4. **What happens to the `accessible-description` of a widget in a collapsed disclosure?** If the disclosure is collapsed, are children still in the AT-SPI tree (they should be, but `Object:state:showing=false`)? Orca by default skips not-showing nodes. So when the user expands a disclosure (Space on focus), we need to move focus *into* it — otherwise the user has no way to know there's content.

5. **How does Slint render a `param_chip` with both a label and a remove button in AT-SPI?** Two AT-SPI nodes (chip text + child button)? One node with two actions? Verify with Accerciser. If two nodes, the user Tabs to the chip (announced "kernel parameter quiet") and then Tabs to the X (announced "remove parameter, button"). If one node, the X is unreachable by keyboard.

6. **Are toast notifications (4 s auto-dismiss) Tab-reachable while visible?** WCAG 2.2.1 (timing) requires user-controllable dismissal of timed content. Spec must say: while a toast is visible, Tab from the page lands on the toast first, then continues. Without that, the [Undo] button is mouse-only.

7. **Does the Live Job Log autoscroll race with screen-reader review-cursor mode?** When Orca is in flat-review mode (Insert+arrows), the user is reading line-by-line. Autoscroll moves the visible region — the user's mental position is now wrong. Recommend: when AT-SPI screen-reader is detected (heuristic: Orca DBus name registered), autoscroll defaults OFF.

## 9. Things that are actually OK

1. **The `accessible-role: dialog` + `accessible-description` commitment in UX_BRIEF §8.** It's incomplete (see §1.1) but it's the right starting point — many apps don't even acknowledge dialog roles. Build on this foundation.

2. **The Mocha contrast against most surfaces.** `--on-surface` on `--surface` is 12.6:1 — well past AAA. The text legibility for sighted users is genuinely strong. Most a11y-claiming apps fail here.

3. **No in-app password fields (`UX_BRIEF.md:13`, `:133`).** This is *enormous* for a11y — a11y of password fields in custom widgets is consistently broken (focus management, paste handling, screen-reader echo). Deferring to polkit-gnome means we inherit GNOME's password-prompt a11y, which is well-tested.

4. **String externalisation (`UX_BRIEF.md:118`).** Mandating no inline literals is the right plumbing for both i18n *and* a11y — `accessible-label` strings flow through the same pipeline, can be translated, and can be audited centrally. Adopt this fully.

5. **Anti-pattern §10.1 forbidding modal-on-modal.** This is a major a11y win. Stacked modals are where focus management goes to die — every nested level adds focus-trap edge cases. The decision to push `[+ New entry]` as an inline panel rather than a modal (UX_MAPPING §B and `GUI_V2_SPEC.md:186`) is the correct application of this rule.

## TL;DR — top 5 most-blocking a11y failures

- **Confirmation Sheet's `accessible-description` is set once and never updated.** Preflight progress, type-to-confirm match, polkit prompt, and live-job-log streaming are all silent to Orca. Without nested live regions, the sheet is unusable for screen-reader users — and the sheet is the single highest-stakes UI in the app.
- **"Enter cancels" breaks keyboard-only intentional workflows.** Cancel-on-Enter collides with Slint's per-button Enter handling and with universal "press Enter in text input to commit" muscle memory; it should be dropped in favour of focus-order + disabled-state-until-typed-confirm.
- **No focus management is spec'd anywhere** — sheet open/close, polkit transition, ActionFooter materialisation, disclosure expansion, inline rename. Every PR will invent its own; PR4's Confirmation Sheet must define the focus-management contract for the entire app or every later flow regresses.
- **Slint may not support `accessible-role: dialog`, live-region re-announcement, modal AT-SPI marking, or `prefers-reduced-motion`.** Five framework unknowns that gate the entire a11y baseline. PR4 must verify before merging — anything else is shipping a promise we can't keep.
- **Focus-ring contrast and primary-button label colours fail WCAG.** 2 px `--accent at 60% alpha` is ~2.7:1 against the darkest card (fails SC 1.4.11), and the unspecified primary-button label colour will default to `--on-surface` against `--accent` fill at 1.4:1 (catastrophic). Both fixable with token-table additions, but mandatory for v2.
